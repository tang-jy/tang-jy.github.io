论chatGPT为什么是神

数据库设计

  • user 包含密码和账户与名称 ✓

  • problem 通过人数,名称,num,是否SPJ,SPJ代码的路径 ✓

  • submission 提交编号,运行结果,时间,关联用户表,题目表 ✓

  • test case 编号,名称,时间内存限制 ✓

  • leaderboard 榜单,显示通过题目数量和罚时,关联用户表 ✓

1. 后端（Backend）逻辑：

a. 用户管理和认证：

  1. 用户注册和登录：
    • 用户在前端注册提供用户名、密码等信息，并通过后端处理进行用户注册。
    • 注册成功后，用户可以使用提供的凭据进行登录。

b. 题目管理和评测：

  1. 题目存储：
    • 后端存储题目信息，包括问题描述、输入输出格式、标准答案等。
  2. 评测引擎：
    • 后端包含评测引擎，负责运行用户提交的代码，并比对输出与标准答案。
    • 可能使用沙盒技术确保评测过程的安全性，防止用户代码对服务器产生不良影响。
  3. 测试用例管理：
    • 后端存储和管理测试用例，用于验证用户提交的代码的正确性。

c. 数据库和持久化：

  1. 用户数据和题目数据存储：
    • 后端使用数据库存储用户信息、题目信息等。
    • 数据库可以包括用户表、问题表、提交记录表等。
  2. 持久化提交记录：
    • 将用户的每次代码提交记录持久化，以便用户查看提交历史和评测结果。

2. 前端（Frontend）逻辑：

a. 用户交互和界面设计：

  1. 用户界面设计：
    • 前端负责设计用户友好的界面，包括用户注册、登录、问题列表、代码编辑器等页面。
  2. 交互设计：
    • 实现用户与系统的交互，例如提交代码、查看问题描述、查看提交历史等。

b. 用户认证和状态管理：

  1. 用户认证：
    • 前端向后端发送用户身份验证请求，获取认证令牌（token）以在后续请求中进行身份验证。
  2. 用户状态管理：
    • 维护用户的登录状态，以便在用户访问受限资源时进行必要的鉴权。

c. 代码编辑器和提交：

  1. 代码编辑器：
    • 提供用户一个代码编辑器，用于编写和修改代码。
  2. 代码提交：
    • 用户在前端界面提交代码，前端将代码提交给后端进行评测。

d. 查看评测结果和历史记录：

  1. 实时评测结果：
    • 用户可以实时查看代码评测的结果，包括通过、未通过等信息。
  2. 提交历史：
    • 用户可以查看之前的提交记录，了解每次提交的评测结果。

3. 交互过程：

  1. 用户与界面交互：
    • 用户通过前端界面与系统进行交互，包括查看问题、提交代码等。
  2. 前后端通信：
    • 前端通过 API 向后端发送请求，请求可能包括登录、注册、提交代码等操作。
  3. 后端处理请求：
    • 后端接收请求，进行相应的处理，包括用户认证、代码评测、数据查询等。
  4. 前后端数据交互：
    • 前后端通过 JSON 或其他数据格式进行数据的传递和交互。

通过这样的前后端逻辑，用户能够在界面友好的 Online Judge 系统中注册、登录、提交代码，并查看评测结果和提交历史。
